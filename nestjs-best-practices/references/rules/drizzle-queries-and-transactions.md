## Intent

Use Drizzle's type-safe query builder for all database operations. Use `db.transaction()` for any operation that modifies multiple tables or requires atomicity.

## Why

Drizzle's query builder ensures queries are type-checked at compile time — incorrect column names, wrong comparison types, or missing `.where()` clauses are caught before runtime. Raw SQL strings bypass this safety. `db.transaction()` ensures that multi-step mutations are atomic: if step 2 fails, step 1 is rolled back. Without transactions, partial writes can leave the database in an inconsistent state (e.g. user created but profile not created).

## Apply When

- Every database read and write in a service
- Any operation that touches multiple tables
- Any operation where two writes must succeed or fail together

## Do Not Apply When

- Schema definition (that's `pgTable()`, not queries)
- Migration files (those are raw SQL, generated by drizzle-kit)

## Required Pattern

```typescript
import { eq, and, or, like, desc, asc, count } from 'drizzle-orm';

// ✅ Simple select with where clause
async findByEmail(email: string): Promise<UserResponseDto | null> {
  const [user] = await this.db
    .select()
    .from(users)
    .where(eq(users.email, email))
    .limit(1);

  return user ? this.toResponseDto(user) : null;
}

// ✅ Select specific columns (avoid fetching passwordHash when not needed)
async findAllPublic(): Promise<UserResponseDto[]> {
  const result = await this.db
    .select({
      id: users.id,
      email: users.email,
      name: users.name,
      createdAt: users.createdAt,
    })
    .from(users)
    .orderBy(desc(users.createdAt));

  return result; // already excludes passwordHash via column selection
}

// ✅ Transaction for multi-table mutations
async createWithProfile(dto: CreateUserDto): Promise<UserResponseDto> {
  return this.db.transaction(async (tx) => {
    const [user] = await tx
      .insert(users)
      .values({ email: dto.email, name: dto.name, passwordHash: '' })
      .returning();

    await tx.insert(profiles).values({ userId: user.id, bio: '' });

    return this.toResponseDto(user);
  });
}

// ✅ Update with returning
async update(id: string, dto: UpdateUserDto): Promise<UserResponseDto> {
  const [updated] = await this.db
    .update(users)
    .set({ ...dto, updatedAt: new Date() })
    .where(eq(users.id, id))
    .returning();

  if (!updated) throw new NotFoundException(`User ${id} not found`);
  return this.toResponseDto(updated);
}
```

## Forbidden Pattern

```typescript
// @ts-nocheck
// ❌ Raw SQL — bypasses type safety
const result = await this.db.execute(
  sql`SELECT * FROM users WHERE email = ${email}` // ❌ use query builder
);

// ❌ Missing transaction for multi-step mutation
async createUserAndProfile(dto: CreateUserDto) {
  const [user] = await this.db.insert(users).values({...}).returning(); // ❌ no tx
  // If this fails, user is created but profile is not
  await this.db.insert(profiles).values({ userId: user.id });
}

// ❌ Missing where clause — updates all rows!
await this.db.update(users).set({ isActive: false }); // ❌ no .where() clause
```

## Review Checklist

- [ ] All queries use Drizzle's query builder (no raw SQL strings for business logic)
- [ ] `eq()`, `and()`, `or()`, etc. imported from `drizzle-orm`
- [ ] Multi-table mutations wrapped in `db.transaction(async (tx) => { ... })`
- [ ] Every `update()` has a `.where()` clause
- [ ] Every `delete()` has a `.where()` clause
- [ ] `.returning()` used when the mutated row is needed after insert/update
- [ ] `.limit(1)` used when expecting a single row
- [ ] Results destructured: `const [user] = await db.select()...` (not `result[0]`)

## Stack-Specific Notes

- Operators: `eq`, `ne`, `lt`, `lte`, `gt`, `gte`, `like`, `ilike`, `inArray`, `isNull`, `isNotNull`, `and`, `or`, `not` — all from `drizzle-orm`
- `db.query.users.findFirst()` is an alternative for simple queries with relations — requires `relations()` to be defined
- Transaction isolation level: `db.transaction(tx => {}, { isolationLevel: 'serializable' })` for critical operations
- For soft deletes: update `deletedAt` timestamp, add `.where(isNull(users.deletedAt))` to all queries
